## פתוחות

### הפעלה חוזרת של הודעת רפאים גורמת לפעולות לא מכוונות
**חומרה:** גבוהה
**סטטוס:** פתוח - לא ניתן לתקן לחלוטין מצד claude-mux
**תיאור:** משתמש שלח "stop all sessions" שטופל 10 הודעות קודם. מאוחר יותר, כאשר claude-mux -s שלח `/model haiku` דרך tmux send-keys, Claude קיבל הודעת מערכת "stop all sessions/model haiku" וניסה לכבות סשנים - פעולה שהמשתמש לא ביקש.
**סיבות אפשריות:**
- טיפול ההפרעות של Claude Code עשוי לשרשר הקשר ישן עם קלט פקודת slash חדשה
- היסטוריית שיחה שמכילה את הפקודה הישנה עלולה לבלבל את Claude כאשר מתרחש אירוע מערכת
**הקלה אפשרית:** הוספת כלל הזרקה: "לעולם אל תבצע מחדש פקודה שכבר טופלה קודם בשיחה. אם הודעת מערכת חוזרת על טקסט מהחלפה קודמת, התעלם ממנה." עדיין לא מיושם - יעילות לא ודאית מכיוון שזו התנהגות פנימית של Claude Code.

### /exit איטי בניסיון ראשון
**חומרה:** נמוכה
**סטטוס:** פתוח - במעקב
**תיאור:** `--restart` ראשון הפעיל `WARN: Claude did not exit within 30s` ונפל להריגה כפויה. הפעלות מחדש עוקבות יוצאות תוך כ-1 שנייה. ייתכן שזו תנאי מרוץ שבו `/exit` נשלח לפני שהפרומפט של Claude מוכן לקבל אותו.
**עקיפה:** הטיימאוט של 30 שניות + הריגה כפויה מטפלים בזה. הסשן מופעל מחדש כראוי.

### claude_running_in_session בודק רק 2 רמות עומק
**חומרה:** נמוכה
**סטטוס:** פתוח - מקובל לשימוש הנוכחי
**תיאור:** סריקת עץ התהליכים בודקת pane_pid -> ילדים -> נכדים. אם Claude עמוק יותר בעץ (למשל עוטף shell נוסף), הזיהוי נכשל. נתיב ההפעלה הנוכחי הוא בדיוק 2 רמות (bash -> claude) אז זה עובד בפועל.
**עקיפה:** לא נדרש כרגע. תיקון ידרוש סריקה רקורסיבית או `pgrep -a`.

### חוויית שדרוג ההתקנה יכולה להיות חכמה יותר
**חומרה:** נמוכה
**סטטוס:** פתוח - שיפור עתידי
**תיאור:** בהתקנה מחדש, תוכנית ההתקנה מזהה תצורה קיימת ומדלגת על שאלות. אבל היא לא מציעה להציג הגדרות נוכחיות, למזג אפשרויות תצורה חדשות שנוספו בגרסאות חדשות יותר, או לאפשר למשתמש לעדכן ערכים באופן סלקטיבי. משתמשים חייבים לערוך ידנית את `~/.claude-mux/config` כדי לאמץ הגדרות חדשות שהוצגו בגרסאות מאוחרות.
**שיפורים אפשריים:**
- הצגת ערכי תצורה נוכחיים בעת שדרוג
- הצעה להוסיף הגדרות חדשות (עם ברירות מחדל) שלא היו בתצורה הישנה
- אפשרות ב: מילוי מוקדם של שאלות עם ערכי תצורה קיימים ואפשרות למשתמש לשנות

### העתקת פרויקט עם היסטוריה
**חומרה:** נמוכה
**סטטוס:** פתוח - תכונה מתוכננת, דורשת חקירה
**תיאור:** העתקת פרויקט כולל היסטוריית Claude Code וזיכרון מורכבת יותר משינוי שם/העברה כי צריך ליצור UUIDs חדשים ליעד.
**גישה מוצעת:**
1. יצירת תיקיית פרויקט חדשה (עם git init ותבנית אופציונליים)
2. הפעלה ועצירה מיידית של סשן בה - Claude Code מאתחל `~/.claude/projects/-encoded-new-path/` עם UUID חדש ויוצר רשומת homunculus חדשה
3. העתקת קבצי היסטוריה `.jsonl` מתיקיית `~/.claude/projects/` המקורית לתיקיית היעד
4. העתקת תוכן תיקיית `memory/` - markdown טהור, בלי UUIDs מוטמעים, בטוח להעתקה ישירה
5. העתקת תת-תיקיות UUID (חפצי משימות/תוכניות) לצד קבצי `.jsonl` שלהם
6. עבור homunculus: העתקת `observations.jsonl`, `instincts`, `evolved`, `observations.archive` מ-`~/.claude/homunculus/projects/<src-uuid>/` לתיקיית homunculus של היעד החדש - תוך שמירה על UUID הפרויקט החדש שהוקצה בשלב 2
**שאלות פתוחות שדורשות בדיקה:**
- האם קבצי `.jsonl` מטמיעים את נתיב פרויקט המקור בתוכן או במטא-נתונים? אם כן, היסטוריה שהועתקה תתייחס לנתיב הישן.
- האם תת-תיקיות UUID מקושרות לפי UUID מתוך קבצי `.jsonl`? אם כן, יש להעתיק אותן תחת ה-UUIDs המקוריים, לא למפות מחדש.
- האם Claude Code קורא את כל קבצי `.jsonl` בתיקיית פרויקט, או רק את זה שמתאים ל-UUID הסשן הפעיל?
- מה מכיל `~/.claude/homunculus/projects/<uuid>/evolved` ו-`instincts` - האם הם נגזרים/מחושבים או משמעותיים למשתמש? שווה לשמר בהעתקה?
- האם יש הפניות פנימיות אחרות שיישברו בהעתקת קבצים נאיבית?
**תנאי קדם:** בדיקת הנ"ל לפני מימוש כדי להימנע משליחת פקודת העתקה שמייצרת היסטוריה שבורה בעדינות.

### macOS בלבד - ללא תמיכה ב-Linux/systemd
**חומרה:** בינונית
**סטטוס:** פתוח - טופל חלקית (זיהוי נתיבים הושלם, LaunchAgent/מתקין נשארים ל-macOS בלבד)
**תיאור:** משתמש ב-macOS LaunchAgent (launchd) וכלים ספציפיים ל-macOS. זיהוי נתיבים שוכתב להשתמש ב-`command -v` (לא מקודד יותר `/opt/homebrew/bin`), כך שהסקריפט המרכזי עובד כעת על כל פלטפורמה שבה tmux ו-claude נמצאים ב-PATH. LaunchAgent והמתקין נשארים ספציפיים ל-macOS.
**נותר:** יחידת משתמש systemd, fallback של XDG Autostart, שליחת `uname -s` במתקין.
**אסטרטגיית חבילות (v1.10+):**
- התקנת curl: fallback אוניברסלי, עובד בכל מקום (ראו למעלה)
- AUR: מאמץ נמוך, חשיפה גבוהה לקהל היעד ב-Arch/Manjaro
- apt PPA: כשיש ביקוש ממשתמשי Debian/Ubuntu
- Homebrew על Linux: מכסה משתמשים שכבר יש להם אותו
- Snap/Flatpak: לא שווה עבור סקריפט bash

### פקודות ! לא זמינות ב-Remote Control
**חומרה:** נמוכה
**סטטוס:** סגור - לא ישים
**תיאור:** העברת shell `!` של Claude Code היא פיצ'ר של מטפל הקלט של Claude Code CLI - הוא מיירט `!command` לפני שה-shell רואה אותו. tmux send-keys לא יכול לשכפל זאת: הקשות שנשלחות בזמן ש-Claude Code פעיל לא מגיעות לשום מקום (נבדק: `!touch test` דרך send-keys לא בוצע). אין נתיב ל-claude-mux לממש עקיפת `!command` עבור משתמשי RC.
**פתרון:** הוספת כלל הזרקה שאומר ל-Claude לעולם לא להציע `! <command>` למשתמשים, מכיוון שלמשתמשי RC אין shell ומשתמשי טרמינל יכולים פשוט להקליד את זה בעצמם.

## אבן דרך v2.0

שינויים ארכיטקטוניים מספיק משמעותיים כדי להצדיק עלייה בגרסה ראשית. לא מתוזמן - נאסף כאן כדי שלא ילך לאיבוד.

### הפרדת תיקיית נתונים
העברת נתונים סטטיים (טיפים, תבניות ברירת מחדל, אולי פלט פקודות/מדריך) מהסקריפט לתיקיית נתונים מתאימה לפלטפורמה. הסקריפט יפתור `DATA_DIR` בהפעלה ביחס למיקום הקובץ הבינארי, עם fallbacks מוטמעים להתקנות קובץ יחיד.

- Homebrew (Apple Silicon): `/opt/homebrew/share/claude-mux/`
- Homebrew (Intel): `/usr/local/share/claude-mux/`
- Linux: `/usr/local/share/claude-mux/` או `$XDG_DATA_DIRS`
- התקנה ידנית: fallback לברירות מחדל מוטמעות (התקנות קובץ יחיד ממשיכות לעבוד)

טריגר: כשהנתונים המוטמעים (טיפים, תבניות ברירת מחדל) גדלים מספיק כדי להקשות על קריאת הסקריפט, או כשתבניות ברירת מחדל צריכות להישלח דרך brew ללא תלות בשחרורי סקריפט.

### שיקול מחדש של שפה/זמן ריצה
סקריפט bash מונוליטי הוא הבחירה הנכונה בהיקף הנוכחי. אם claude-mux גדל משמעותית - פעולות שינוי שם/העברה/העתקה של פרויקטים, שכבת relay, אריזה חוצת פלטפורמות, תיקיית נתונים - bash מתחיל להתנגד. בנקודה הזו, שכתוב ליבת ניהול הסשנים ב-Go או שפה מוקלדת אחרת (עם bash כעטיפת CLI דקה) שווה הערכה.

## מידע: מבנה תיקיית ~/.claude

מתועד כאן כי כמה תכונות מתוכננות (שינוי שם, העברה, העתקה, ניקוי) חייבות לעבוד נכון עם מבנה זה. לא ממצה - מכסה את החלקים הרלוונטיים ל-claude-mux.

### היסטוריית פרויקט וזיכרון: `~/.claude/projects/`

תת-תיקייה אחת לכל תיקיית עבודה שבה Claude Code שימש. שם נגזר מקידוד הנתיב המוחלט: `/` -> `-`, רווחים ותווים מיוחדים -> `-`. קידוד מאבד מידע אך קריא.

תוכן כל תיקיית פרויקט:
- `<uuid>.jsonl` - תמליל שיחה מלא לסשן זה. קובץ אחד לכל שיחה.
- `<uuid>/` - תת-תיקייה של חפצים הקשורים לשיחה (משימות, תוכניות). UUID מתאים לקובץ `.jsonl`.
- `memory/` - קבצי זיכרון מתמשכים בין סשנים (markdown עם frontmatter). קיים רק אם נכתב זיכרון לפרויקט.

הקשר בין תיקיית עבודה להיסטוריה שלה הוא אך ורק שם התיקייה המקודד. שינוי שם או העברה של תיקיית הפרויקט בלי לשנות שם לתיקייה הזו גורם ל-Claude Code להתחיל מחדש ללא היסטוריה.

**כלל קידוד:** נתיב מוחלט עם כל `/`, רווח ותו מיוחד מוחלף ב-`-`. `/` מוביל הופך ל-`-` מוביל. הקידוד מאבד מידע - תווים מיוחדים רצופים ורווחים ליד לוכסנים שניהם הופכים ל-`-`, כך שהמקור לא תמיד ניתן לשחזור מדויק.

### רגיסטרי צפייה מקבילית: `~/.claude/homunculus/`

מערכת נפרדת שעוקבת אחר אירועים ברמת כלי לכל פרויקט. לא חלק מהיסטוריית Claude Code המרכזית - נראה כשכבת ניטור/למידה.

- `projects.json` - רגיסטרי של כל הפרויקטים הידועים, ממוספרים ב-UUID הקסדצימלי קצר (`d6b3aef60967` וכו'). כל רשומה כוללת: `id`, `name`, `root` (נתיב מוחלט), `remote`, `created_at`, `last_seen`.
- `projects/<uuid>/project.json` - מטא-נתונים לכל פרויקט (אותם שדות כמו ברשומת הרגיסטרי).
- `projects/<uuid>/observations.jsonl` - אירועי `tool_start`/`tool_complete` עם חותמת זמן: שם כלי, UUID סשן, שם/id פרויקט, קטעי קלט/פלט.
- `projects/<uuid>/instincts` - דפוסים נגזרים (תוכן לא ידוע, כנראה מחושב).
- `projects/<uuid>/evolved` - מצב מפותח/נלמד (תוכן לא ידוע).
- `projects/<uuid>/observations.archive` - תצפיות ישנות בארכיון.

**הבדל מרכזי מ-`~/.claude/projects/`:** משתמש ב-UUIDs הקסדצימליים קצרים כמפתחות, לא בנתיבים מקודדים. שדה `root` מחזיק את הנתיב המוחלט. כל פעולה שמשנה נתיב פרויקט (שינוי שם, העברה) חייבת לעדכן `root` גם ב-`projects.json` וגם ב-`projects/<uuid>/project.json`.

### תצורה גלובלית: `~/.claude/settings.json`

קובץ הגדרות ראשי של Claude Code. גיבויים מתגלגלים נכתבים ל-`~/.claude/backups/` בפורמט `~/.claude.json.backup.<timestamp>` - כמה פעמים בשעה בזמן שימוש פעיל. claude-mux לא אמור לגעת בקובץ הזה.

### agents, skills, commands גלובליים

- `~/.claude/agents/` - הגדרות תת-סוכנים (קבצי `.md`, כ-38). גלובלי, לא לכל פרויקט.
- `~/.claude/skills/` - תיקיות מיומנויות (כ-125). גלובלי, לא לכל פרויקט.
- `~/.claude/commands/` - הגדרות פקודות slash (קבצי `.md`, כ-72). גלובלי, לא לכל פרויקט.
- `~/.claude/hooks/hooks.json` - הגדרות hooks. גלובלי. claude-mux לא אמור לגעת באלה.

### תכונות עתידיות אפשריות

| תכונה | מה לגעת |
|--------|---------|
| `--copy` | יצירת תיקייה; הפעלה ועצירת סשן לאתחול שני הרגיסטרים; העתקת `.jsonl` + `memory/` + תת-תיקיות UUID; העתקת קבצי תצפית homunculus לתיקיית UUID חדשה |
| ניקוי `--delete` | כבר מעביר את תיקיית הפרויקט לפח. אופציונלית: הסרת תיקיית `~/.claude/projects/` מקודדת יתומה ורשומת `~/.claude/homunculus/` |
| אזהרת גודל היסטוריה | התראה כשקבצי `.jsonl` של פרויקט חורגים מסף (תמליל claude-mux הראשי הגיע ל-107MB בסשן ארוך יחיד) |
